### Site 1 — rap-public-site (Netlify: `rap-designer-programs`)

**Lo que ya está en el repo** (untracked, en working dir de Adrian):

- `rap-public-site/netlify/functions/_supabase.js` — clon del cliente service-role de designer-plan-site.
- `rap-public-site/netlify/functions/lead-capture.js` — webhook handler único para los dos forms.

**Qué hace `lead-capture.js`:**

Recibe el webhook `submission-created` de Netlify Forms. Discrimina por `form_name`:

| `form_name` | Source registrado | Lista a la que se agrega |
|---|---|---|
| `whitepaper-request` | `whitepaper-request` | `designers-all` |
| `designer-sign-up` | `designer-sign-up` | `designers-all` |

Para cada submission válida:
1. Upsert en `leads` por `email` (citext), con `consent_at = now()` y `consent_text` anclado al texto literal del HTML actual.
2. Insert en `lead_events` con `event_type='form_submitted'`, `source=<form_name>`, `payload` jsonb con submission completa + IP + user_agent + netlify_submission_id.
3. Upsert en `email_list_members` para la lista `designers-all`.

Manejo defensivo: honeypot drop silencioso, form_name desconocido devuelve 200 (sin retry), email inválido devuelve 400, Supabase caído devuelve 500 (pero Netlify Forms sigue guardando el backup en su dashboard).

## Cambios en schema de Supabase

**Cero.** Esta ronda usa el schema existente tal cual. No hay migraciones nuevas.

Recordatorio del schema vigente (definido en `supabase/migrations/20260511_base_schema.sql` + `20260512_designer_plan_extensions.sql`):

- `leads` — una fila por persona, key = email citext, con `consent_at` + `consent_text`.
- `lead_events` — append-only, todos los eventos por lead.
- `partners` — 1:1 con `leads`, status pending/approved/rejected/suspended.
- `plans` — 3 filas seed (tier-one, tier-two, tier-three).
- `orders` — purchases (todavía nadie escribe acá; Stripe real lo hará en otra ronda).
- `email_lists` + `email_list_members` — listas curadas con 4 filas seed (`designers-all`, `designers-partners`, `designers-pending`, `clients-direct`).
- `admin_leads_view` — view que joins leads + partner + last event para la UI del marketing-center.

## Smoke tests (SQL listo para pegar en Supabase Studio)

### Después del deploy de Site 1
```sql
-- 1. Lead recién creado por una submission de whitepaper o sign-up
select id, email, full_name, company, consent_at, created_at
from public.leads
order by created_at desc
limit 5;

-- 2. Evento asociado con metadata completa
select source, event_type, created_at,
       payload->>'netlify_submission_id' as nf_id,
       payload->>'ip' as ip
from public.lead_events
order by created_at desc
limit 5;

-- 3. Membresía a 'designers-all'
select l.email, el.slug, m.added_at
from public.email_list_members m
join public.leads l       on l.id = m.lead_id
join public.email_lists el on el.id = m.list_id
where el.slug = 'designers-all'
order by m.added_at desc
limit 5;
```

### Después del smoke test de Site 2
```sql
-- Distribución de sources — debe haber 'landing_popup', 'partner_application', 'cart_started'
select source, count(*), max(created_at) as last_seen
from public.lead_events
group by source
order by last_seen desc nulls last;

-- Partners pending creados por /partner-apply
select p.id, p.status, p.referral_code, l.email, l.company
from public.partners p
join public.leads l on l.id = p.lead_id
where p.status = 'pending'
order by p.created_at desc
limit 10;
```
